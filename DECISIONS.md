# 설계 결정 이력 (Architecture Decision Records)

본 문서는 시스템 구축 과정에서 고민했던 핵심 기술적 판단과 선택의 근거, 트레이드오프를 기록한 설계 결정 이력 문서입니다. 모든 결정은 실제 k6 부하 테스트로 측정한 지표와 실증적인 동시성 제어 결과를 기반으로 검증되었습니다.

---

## 기술 스택 구성

- **Language & Framework**: Java 17 + Spring Boot 3.5
- **Database**: MySQL 8.0 (영속성 저장)
- **Caching & Coordination**: Redis 7.0 (실시간 재고 소스, Look-aside 캐시, 분산 락)
- **Infrastructure**: Docker Compose + nginx (2개 애플리케이션 인스턴스에 대한 로드 밸런싱)
- **Performance Evaluation**: k6 (부하 및 정합성 테스트)

---

## 쟁점 1. 재고 정합성을 어디서 보장할 것인가

### 상황
선착순 10개 한정 수량 상품에 순간적으로 1,000 TPS 이상의 고밀도 트래픽이 집중되는 환경입니다. 재고가 초과 판매되는 일이 단 1건도 없어야 하며, 고동시성 상황에서도 높은 응답 속도를 유지해야 하는 기술적 요구가 있습니다.

### 선택지
- **선택지 A**: RDBMS(MySQL)의 비관적 락(`SELECT ... FOR UPDATE`) 또는 낙관적 락(`@Version`) 활용
- **선택지 B**: **Redis Lua 스크립트를 통한 단일 원자 연산으로 인메모리 재고 차감 처리 (최종 선택)**

### 왜 그렇게 판단했는지
- **비관적 락(선택지 A)**은 강력한 정합성을 보장하지만, 동일한 로우(Row)에 수백 개의 요청이 동시 진입할 경우 데이터베이스의 로우 락 경합으로 인해 쓰레드들이 대기하게 되고, 처리량이 급격하게 저하됩니다. 낙관적 락 또한 충돌 시 잦은 재시도로 인해 데이터베이스 리소스를 심각하게 소모합니다.
- **Redis Lua 스크립트(선택지 B)**는 싱글 쓰레드로 동작하는 Redis 엔진 내부에서 스크립트 전체가 하나의 원자적 명령처럼 수행됩니다. '재고 조회(check) 후 차감(act)'하는 일련의 과정이 다른 명령의 개입 없이 원자적으로 보장되므로, 데이터 유실이나 초과 판매 위험이 극히 낮습니다. 또한 디스크 I/O 없이 인메모리로만 연산이 수행되므로 초당 수천 건 이상의 처리를 무리 없이 통과시킵니다.
- **트레이드오프**: Redis가 시스템의 핵심 SPOF(Single Point of Failure)가 됩니다. 이를 극복하기 위해 운영 환경에서는 Redis Sentinel이나 Redis Cluster와 같은 고가용성 구조를 추가로 구비하는 비용이 수반됩니다. 본 시스템에서는 정합성과 성능을 최우선 가치로 두어 Redis Lua 스크립트 방식을 기본 설계로 채택하였습니다.
- **실증 데이터**: 동시 100요청의 경합 테스트에서 `bookings` 생성 수는 정확히 10건으로 제어되었으며, 초과 판매율 0.00%를 달성하였습니다.

## 쟁점 2. 가예약(Reserve) → 확정(Confirm) 2단계 흐름의 단순화

### 상황
일반적인 선착순 커머스 시스템에서는 재고를 선점하는 '가예약' 단계와 실제 대금 지불 후 완료되는 '확정' 단계의 2단계 라이프사이클을 가집니다. 하지만 이 복잡한 분리 방식이 시스템 구성과 자원에 주는 오버헤드를 고려하여 대안을 고민해야 했습니다.

### 선택지
- **선택지 A**: 가예약(Pending) 상태 저장 후 일정 시간 미결제 시 스케줄러로 재고를 되돌리는 2단계 분리 방식
- **선택지 B**: **1단계 동기 결제 처리 방식 (Redis Lua 재고 차감 + 외부 PG 동기 승인 성공 시 즉시 DB 확정) 및 실패 시 Saga 보상 처리 (최종 선택)**

### 왜 그렇게 판단했는지
- 본 시스템의 시나리오는 사용자가 결제 페이지에서 대기하며 즉시 승인을 받는 실시간 동기식 흐름입니다. "가예약 후 수십 분간 대기"와 같은 비동기 유예 단계의 필요성이 상대적으로 낮습니다.
- **2단계 방식(선택지 A)**을 취할 경우, 만료 상태를 관리하기 위해 분산 스케줄러와 미결제 예약 해제 데몬을 상시 구동해야 하며, 데이터베이스 트랜잭션 경합과 테이블 락 점유 시간이 늘어나 부하가 가중됩니다.
- **1단계 방식(선택지 B)**은 Lua 스크립트로 신속하게 실시간 재고를 차감한 직후, 외부 결제 대행사(PG) 호출이 성공하면 데이터베이스에 즉시 `CONFIRMED` 예약을 저장하는 고속 파이프라인을 구축할 수 있습니다. 결제 도중 실패가 발생하면 Saga 보상 트랜잭션 로직을 태워 재고와 차감된 포인트를 안전하게 원복합니다.
- **트레이드오프**: 결제 성공 후 예기치 못한 애플리케이션의 비정상 종료(JVM 강제 종료 등)가 발생하는 극단적인 예외 상황에서는 일시적으로 재고가 선점된 상태로 묶일 수 있습니다. 이는 시스템 배치나 미완결 예약 정리 백그라운드 태스크로 잔여분을 정리하는 방어책으로 해결 가능합니다.

## 쟁점 3. 멱등성(Idempotency) — 짧은 간격의 중복 결제 차단 방식

### 상황
사용자가 결제 버튼을 연속으로 클릭하거나(Double Click), 모바일 네트워크 환경 불안정으로 동일한 요청 패킷이 중복으로 백엔드에 밀려드는 현상이 빈번히 발생합니다. 결제 및 예약은 절대 중복 발생해서는 안 되며, 최단 시간 내에 동일 요청을 무해하게 걸러내야 합니다.

### 선택지
- **선택지 A**: 별도의 전용 `idempotency_keys` 관리 테이블을 생성하고, 매 요청마다 이 테이블에 상태를 쓰고 읽는 방식
- **선택지 B**: **3-tier 멱등 방어선 (1차: bookings 테이블 직접 조회, 2차: Redis SETNX 분산 락, 3차: 데이터베이스 unique constraint) 구축 (최종 선택)**

### 왜 그렇게 판단했는지
- **전용 관리 테이블 활용(선택지 A)**은 모든 중복 검사 요청이 영속 데이터베이스 쓰기/조회 병목을 일으키며 전체 응답 성능을 하락시키는 요인이 됩니다.
- **3-tier 방어선(선택지 B)**은 다음과 같이 자원 효율성과 안전성을 계층화합니다:
  1. **1차 방어**: 이미 이전에 완료된 요청이 재인입되면 `bookings` 테이블의 고유 인덱스 조회를 통해 데이터베이스에서 즉시 기 생성된 예약 DTO를 그대로 돌려줍니다. 인덱스 기반 단건 조회의 레이턴시는 매우 짧습니다 (p95 4.9ms).
  2. **2차 방어**: 거의 동시에 두 개 요청이 도달하는 경우(In-flight 상태), Redis `SETNX idempotency:lock:{key}` (TTL 10초)를 사용하여 동일 쓰레드 진입을 메모리 상에서 차단합니다.
  3. **3차 방어**: 2차 분산 락이 예기치 않게 해제되거나 Redis 일시 장애 시 발생할 수 있는 극단적인 경합에 대비하여, `bookings.idempotency_key` 컬럼 자체에 `UNIQUE` 물리 제약을 부여합니다. JPA 영속화(`saveAndFlush`) 시점에 충돌이 발생하면 즉시 예외로 처리하여 복구합니다.
- **트레이드오프**: 요청 바디(Body)의 페이로드가 다름에도 동일한 멱등성 키를 보낸 극단적 부정 케이스를 잡아내는 해싱 비교 로직(`request_hash` 검증)은 생략되었습니다. 이는 차후 1차 방어선 역할을 전용 테이블이나 정교한 해시 필터로 전환할 때 쉽게 확장할 수 있도록 코드를 설계해 두었습니다.

## 쟁점 4. 결제 수단 확장성 — 신규 수단 추가 시 로직 변경 최소화

### 상황
초기 설계에서는 신용카드(`CREDIT_CARD`), 포인트(`Y_POINT`), 간편결제(`Y_PAY`)의 세 가지 수단이 존재하나, 향후 다양한 외부 페이 수단이나 가상자산 등 새로운 수단이 자유롭게 도입될 수 있어야 합니다. 신규 수단이 추가될 때 핵심 예약 생성 비즈니스 로직에 영향이 전혀 없거나 최소화되어야 합니다.

### 선택지
- **선택지 A**: `BookingService` 내부에서 결제 타입별로 `if/else` 분기하여 결제를 처리하는 절차지향적 방식
- **선택지 B**: **디자인 패턴(Strategy, Composite) 및 Java sealed interface/record 문법을 조합한 구조적 추상화 (최종 선택)**

### 왜 그렇게 판단했는지
- **분기 처리 방식(선택지 A)**은 단기적으로 직관적이나, 새로운 수단이 추가될 때마다 거대한 조건문 블록을 수정해야 하므로 객체지향의 개방-폐쇄 원칙(OCP)을 심각하게 위반하고 버그 가능성을 높입니다.
- **추상화 방식(선택지 B)**을 적용하여 각 결제 수단을 `PaymentMethod` 인터페이스로 독립시켰으며, 이를 총괄하는 `PaymentProcessor`를 컴포지트 패턴으로 구성했습니다. 또한 결제 세부 정보의 안정적인 컴파일러 검증을 위해 `sealed PaymentDetail` 인터페이스와 하위 `record`들을 정의했습니다.
- 이를 통해 신규 결제 수단을 추가할 때:
  1. `PaymentMethod` 인터페이스를 상속받은 새 클래스 구현
  2. `PaymentDetail` 하위 레코드 추가
  3. `PaymentType` enum에 추가
  이 외의 기존 비즈니스 흐름 코드에는 단 한 줄의 수정도 발생하지 않아 시스템 안정성이 유지됩니다.

## 쟁점 5. 복합 결제 부분 실패 시 보상 트랜잭션 (Saga)

### 상황
"신용카드 14.9만 원 + 포인트 1만 원"과 같은 복합 결제가 처리되는 도중, 포인트 차감은 성공했으나 이후 외부 PG 연동 단계에서 잔액 부족이나 한도 초과 등으로 신용카드 결제가 거절될 수 있습니다. 분산된 결제 트랜잭션의 일부만 성공한 경우 이를 원격으로 안전하게 복원하는 메커니즘이 필요합니다.

### 선택지
- **선택지 A**: 분산 트랜잭션 기술(2PC, XA)을 도입하여 전체 공유 자원의 트랜잭션을 동기식으로 일괄 커밋/롤백 처리
- **선택지 B**: **Saga 패턴의 보상 트랜잭션 기반 LIFO(Last-In, First-Out) 역순 롤백 수행 (최종 선택)**

### 왜 그렇게 판단했는지
- **2PC/XA 기술(선택지 A)**은 네트워크 상의 조율자 오버헤드가 매우 크며, 타사 외부 PG API는 XA 트랜잭션 프로토콜에 참여할 수 없으므로 현실적으로 도입이 불가능합니다.
- **Saga 패턴(선택지 B)**을 채택하여 개별 단계는 즉시 독립된 트랜잭션으로 커밋하고, 도중 실패가 감지되면 역순으로 성공했던 결제 건들의 환불 API를 차례대로 호출하는 방식을 적용했습니다.
- **결제 실행 순서 최적화**: 상대적으로 비싸고 실패 확률이 높은 외부 연동 채널(`CREDIT_CARD`, `Y_PAY`)을 가장 나중에 실행하고, 성공 확률이 높고 취소가 용이한 내부 포인트 채널(`Y_POINT`)을 먼저 실행하도록 우선순위를 부여했습니다.
- **보상 트랜잭션의 멱등성 확보**: 포인트 환불 메서드(`restore()`)는 이미 환불되었는지를 데이터베이스에서 `existsByOriginalTransactionIdAndType`로 선제 조회하여 중복 환불 사고를 사전에 효과적으로 방지합니다.

## 쟁점 6. 결제 통합 시의 트랜잭션 경계 설정 (B-3 방식)

### 상황
외부 PG 승인 API 호출은 일반적으로 200ms에서 길게는 수 초 이상 소요되는 원격 네트워크 I/O 작업입니다. 이 작업을 스프링의 일반적인 데이터베이스 `@Transactional` 지점 내부에 포함할 경우, 외부 응답을 기다리는 동안 HikariCP 커넥션 풀의 커넥션을 반환하지 않고 쥐고 있게 되어 순식간에 커넥션 고갈 및 전체 API 마비 현상을 초래합니다.

### 선택지
- **선택지 A**: `BookingService.createBooking()` 전체 메서드에 `@Transactional`을 선언하여 단순하게 묶는 구조
- **선택지 B**: **외부 I/O는 트랜잭션 바깥에서 수행하고, 데이터베이스 작업만 국소 트랜잭션으로 분리하는 구조 (B-3 방식) (최종 선택)**

### 왜 그렇게 판단했는지
- **전체 트랜잭션 묶기(선택지 A)**는 대규모 부하 상황에서 커넥션 풀 고갈의 가장 큰 원인입니다. 성능 병목 지점이 데이터베이스가 아닌 커넥션 풀이 되어 서비스 전체가 먹통이 됩니다.
- **트랜잭션 바깥 배치(선택지 B)**를 설계하여 다음과 같이 흐름을 엄격하게 통제했습니다:
  1. 외부 PG 승인과 Redis 락 및 재고 차감 등은 데이터베이스 트랜잭션이 전혀 없는 상태에서 독립 실행합니다.
  2. 포인트 차감은 `PointBalanceAdapter` 내부의 짧은 전용 트랜잭션 안에서 수행합니다.
  3. 모든 외부 승인이 온전하게 성공한 후, 최종 예약 영속화 작업만 `BookingPersistenceService`에서 `@Transactional`로 묶어 밀리초 단위의 극히 짧은 순간에 커넥션을 획득하고 릴리즈합니다.
- **트레이드오프**: 최종 영속화 단계에서 불의의 서버 장애 등으로 영속화가 실패할 경우, 결제 승인은 났는데 예약 정보가 남지 않는 비일관성이 발생할 수 있습니다. 이는 catch 블록 내에서 결제 대행사 환불 API를 동기적으로 호출하는 견고한 에러 핸들링과, 미완결 건을 추적하는 스케줄러 설계로 안전망을 구축하여 방어할 수 있습니다.

## 쟁점 7. BookingService의 책임 분리 및 self-call 우회

### 상황
멱등 락 획득, 재고 차감, 유효성 검사, 결제 및 예외 시 보상 로직이 하나의 `BookingService`에 모이게 되면 클래스 코드가 비대해져 유지보수가 어렵습니다. 또한, 특정 내부 메서드에 트랜잭션을 적용하려고 할 때 스프링 프록시의 `self-call` 제약으로 인해 `@Transactional`이 무시되는 아키텍처적 위험에 직면하게 됩니다.

### 선택지
- **선택지 A**: 단일 클래스로 뭉쳐두고 `AopContext.currentProxy()` 등을 사용해 내부 self-call 문제를 우회하는 방식
- **선택지 B**: **비즈니스 책임을 5개의 세부 빈(Bean)으로 파편화 및 캡슐화하여 SRP(단일 책임 원칙)를 준수하고 의존성을 주입받아 호출하는 방식 (최종 선택)**

### 왜 그렇게 판단했는지
- **AOP 우회 기술(선택지 A)**은 프레임워크 의존도가 지나치게 높아지고 가독성이 떨어지며, 테스트 시 Mocking하기에 최악의 구조가 됩니다.
- **5개 전용 컴포넌트 분리(선택지 B)**를 통해 역할을 명확히 쪼갰습니다:
  - `BookingService`: 전체 오케스트레이션 및 Saga 흐름 제어 (트랜잭션 없음)
  - `BookingPersistenceService`: 데이터베이스 영속화 단독 담당 (물리 트랜잭션 적용)
  - `RedisStockManager`: Redis Lua 스크립트 실행 및 롤백 제어
  - `IdempotencyLockManager`: 멱등 분산 락 통제
  - `PaymentCommandMapper`: 요청 바디 구조를 내부 결제 명령으로 안전하게 변환
- 이를 통해 `BookingPersistenceService`를 주입받아 호출함으로써 프록시를 정상적으로 거치게 되어 `self-call` 부작용을 원천 배제하였고, 단위 테스트 작성 및 결함 추적이 비약적으로 용이해졌습니다.

## 쟁점 8. Checkout API의 조회 성능 개선을 위한 캐싱 전략

### 상황
선착순 이벤트 직전에는 사용자들이 폭발적으로 새로고침을 연타하며 주문서 조회 API(`GET /checkout`)를 호출합니다. 모든 사용자가 특정 상품 ID를 타겟팅하므로 데이터베이스 단일 로우에 대한 집중 쿼리 병목(Hotspot)이 발생하여 CPU 부하와 인프라 정체가 가중됩니다.

### 선택지
- **선택지 A**: 사용자의 요청 때마다 항상 RDBMS(MySQL) 조회를 수행하고, DB 인프라 사양을 고스란히 끌어올리는 스케일업(Scale-up)
- **선택지 B**: **변하지 않는 상품 기본 메타(Product Info)에 대해 Redis Look-aside 캐시(TTL 5초)를 적용하고, 변동성이 심한 개인 자산(User Wallet)은 캐시에서 배제하는 하이브리드 캐싱 (최종 선택)**

### 왜 그렇게 판단했는지
- **전체 DB 조회(선택지 A)**는 부하 테스트 결과, 순간적인 spike 부하(1,000 TPS) 상황에서 p95 응답 지연 시간이 **208ms**까지 치솟고 수백 건의 응답 누락이 발생하는 한계가 실증되었습니다.
- **하이브리드 캐싱(선택지 B)**은 현명한 선택입니다:
  - **상품 기본 정보**: 가격, 오픈 시간, 객실 설명 등은 매우 정적인 정보이므로 5초 동안 실시간 정보와 약간의 편차(Stale)가 생기더라도 비즈니스적 임팩트가 적습니다. 캐시를 적용하면 60,000번 유입될 DB 쿼리가 단 12번의 DB 조회로 압축됩니다.
  - **사용자 포인트 정보**: 사용자 지갑은 개인마다 철저히 격리되어 있어 단일 로우 집중 병목이 생기지 않는 자연 분산 구조입니다. 잔여액의 실시간 정확도가 매우 민감한 자산이므로 캐싱하지 않고 무조건 실시간 조회를 고수합니다.
- **실증 데이터**: 하이브리드 캐싱 전략을 도입한 결과, 동일한 1,000 TPS 부하 조건에서 p95 레이턴시가 **5.6ms**로 비약적으로 단축되었으며 요청 유실률은 **0%**로 떨어졌습니다.

## 쟁점 9. Redis 장애 시의 복원 및 Fallback 안전성 (설계 사양)

### 상황
본 시스템의 실시간 재고 관리와 멱등 분산 락 처리가 전적으로 Redis에 의존하고 있어, 만약 네트워크나 메모리 고갈 등으로 Redis 클러스터가 일시 불능이 될 때의 아키텍처적 대응 전략이 필수적입니다.

### 선택지
- **선택지 A**: Redis 장애 발생 시 MySQL 비관적 락으로 우회하여 계속해서 신규 예약을 수락하는 폴백 방식
- **선택지 B**: **Fail-Fast(정합성 최우선 수호) 전략 하에 Circuit Breaker를 활성화하여 즉시 안전 에러(503)를 리턴하는 방식 (최종 선택)**

### 왜 그렇게 판단했는지
- **DB 비관적 락 폴백(선택지 A)**은 이상적이어 보이지만 ...
- **Fail-Fast 및 Circuit Breaker(선택지 B)**가 ...
- ...

### 현재 구현 상태
- **본 과제 시간 내 Circuit Breaker(Resilience4j)는 미구현 상태**입니다. 현재는 Redis 장애 시 RedisStockManager에서 예외가 발생하면 BookingService의 catch 블록으로 진입하여 보상 흐름이 수행된 후 500 응답이 반환됩니다. 정합성은 깨지지 않지만 fail-fast(503)는 아닙니다. 운영 환경 도입 시 `resilience4j-spring-boot3`로 보강 가능합니다.

## 쟁점 10. 수평 확장을 통한 피크 트래픽 대응성 확보

### 상황
로컬 단일 컨테이너로 테스트 시 애플리케이션의 단일 CPU 코어 및 메모리 자원 한계로 1,000 TPS 유입 시 높은 타임아웃과 패킷 드롭이 필연적으로 계측되었습니다. 추가적인 하드웨어 도입 없이 아키텍처 구조의 조정을 통해 피크 부하 유입을 흡수해야 합니다.

### 선택지
- **선택지 A**: 단일 애플리케이션 서버 사양을 높이는 물리적 스케일업(Scale-up)
- **선택지 B**: **Docker Compose로 복수의 독립 인스턴스(app1, app2)를 띄우고 nginx 로드 밸런서로 부하를 세분화하는 수평 확장(Scale-out) (최종 선택)**

### 왜 그렇게 판단했는지
- **스케일업(선택지 A)**은 물리적인 임계 비용이 기하급수적으로 커지며, 서버 단일 장애점(SPOF)을 제거하지 못해 서버 다운 시 서비스 전체 중단으로 이어집니다.
- **수평 확장(선택지 B)**은 저렴한 소형 가상 컨테이너들을 다수 증설하여 유연하게 트래픽 분할 처리가 가능합니다. Nginx를 통해 인스턴스 간 물리적 독립성이 유지되므로 특정 서버가 불능이 되더라도 무중단 운영이 가능합니다.
- 복수 인스턴스 정합성 조율은 분산 메모리 저장소인 단일 Redis 서버와 단일 MySQL을 공유하므로, 인스턴스 수가 무한히 늘어나더라도 동시성 락 충돌 감지 및 재고 차감의 정합성이 일관되게 유지됩니다.
- **실증 데이터**: 단일 컨테이너 대비 분산 수평 확장 도입 후, Booking Spike 1,000 TPS 시나리오에서 dropped 건수가 **11,487건에서 단 18건**으로 개선되었으며 p95 응답 속도가 **5,213ms에서 22.8ms**로 단축되었습니다.

## 쟁점 11. 데이터 시드(Seed) 시의 멱등성 및 안전성 보장

### 상황
애플리케이션이 구동될 때 성능 테스트 및 정상 작동을 위해 일정 수 이상의 데이터 초기 시딩(`ProductDataInitializer`)이 동작해야 합니다. 단, 수평 확장으로 인해 두 개 이상의 애플리케이션 인스턴스가 동시에 기동되는 특성 상, 시드 삽입 로직이 중복 호출되어 데이터가 오염되거나 PK 충돌 예외가 발생하며 기동 자체가 취소될 위험이 존재합니다.

### 선택지
- **선택지 A**: 시딩 로직의 에러를 무시하고 덮어쓰거나 무조건 예외 처리로 던지는 단순 구성
- **선택지 B**: **Redis 기반 분산 락(`SET NX EX 10`) 및 기 존재 카운트 더블 체크 조합을 활용한 다중 기동 보호 장치 (최종 선택)**

### 왜 그렇게 판단했는지
- **단순 처리(선택지 A)**는 기동 오류 로그로 인해 인프라 모니터링 경보를 상시 유발하며, 원장 데이터를 중복 주입하여 테스트 신뢰도를 무너뜨립니다.
- **분산 락과 더블 체크(선택지 B)**의 작동 흐름은 다음과 같습니다:
  1. 기동 시 두 인스턴스가 동시에 `ProductDataInitializer`를 구동하면, Redis에 `lock:data_seed` 키를 선점 획득하도록 경쟁시킵니다.
  2. 락을 선점한 1등 인스턴스만 실제 데이터베이스 `products.count()`를 조회하여 비어있음을 확인한 후, 깨끗하게 데이터 주입 트랜잭션을 완수합니다.
  3. 락 획득 경쟁에서 밀린 2등 인스턴스는 일정 시간 폴링(재시도)을 거친 후, 데이터가 이미 성공적으로 채워졌음을 인지하고 삽입을 평화적으로 Skip합니다.
  4. 단, In-Memory에 있는 Redis 재고 값(`stock:product:1`)만큼은 휘발 방지를 위해 두 인스턴스 모두 웜업 갱신을 허용합니다(갱신은 멱등하므로 안전합니다).
- 이를 통해 이중 기동 상황에서도 정합성이 유지되며, 기동에 단 한 번의 예외 로그도 남지 않는 깨끗한 로컬 데브옵스 인프라 환경을 완성했습니다.
