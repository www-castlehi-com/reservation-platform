# Booking + Payment 통합 부하 테스트 및 단계별 튜닝 리포트

본 보고서는 결제 기능 도입 이전과 이후의 성능 지표를 진단하고, 발견된 병목 요인에 대한 개선 가설을 수립하여 실제 최적화(우선순위 1)를 통해 성능을 극대화한 과정을 서사적으로 다룬 종합 리포트입니다.

---

## 1. 테스트 환경 및 사전 조건

- **인스턴스 구성**: 단일 로컬 인스턴스 (Spring Boot 1대 + MySQL 8 + Redis 7-Alpine)
- **대상 API**: `POST /bookings` (카드 + 포인트 복합 결제)
- **HikariCP**: 기본값 (maximum-pool-size=10)
- **Tomcat 스레드**: 기본값 (threads.max=200)
- **구현 스펙**: Redis Lua 재고 차감 + 3-tier 멱등 + Composite Payment (Saga LIFO 보상) + B-3 트랜잭션 분리
- **사전 조건**: productId=1, 재고 10개 (`redis-cli SET stock:product:1 10`)

---

## 2. [기존 비교] 결제 기능 통합 전후의 성능 진단

결제 기능이 없는 가벼운 예약 로직(Day 2)에 실제 복합 결제(카드 외부 PG 연동 + 포인트 차감 및 영속화)를 통합한 직후(Day 3 - 튜닝 전)의 성능 비교 결과입니다.

### 📊 1차 측정 결과 (결제 통합 직후 - 무튜닝 상태)

| 시나리오 | 항목 | 결제 없음 (Day 2) | 결제 포함 (Day 3 - 튜닝 전) | 성능 변화 |
|---|---|---|---|---|
| **stress**<br>(동시 100요청, 재고 10) | TPS | 234 | 232 | -1% (변화 미미) |
| | p95 latency | 415ms | 415ms | 동일 |
| | DB bookings 결과 | 정확히 10건 | 정확히 10건 | ✅ 정합성 유지 |
| **idempotency**<br>(같은 키 100회 × 3) | TPS | 627 | 478 | -24% (성능 저하 시작) |
| | p95 latency | 4.9ms | 3.8ms | -22% |
| | DB bookings 결과 | 정확히 3건 | 정확히 3건 | ✅ 정합성 유지 |
| **spike**<br>(50→1000→50, 1.5분) | **평균 TPS** | **684** | **454** | 🔴 **-34% 급락** |
| | **p95 latency** | **50.6ms** | **5,213ms** | 🔴 **+10,200% 폭증 (5.2초 지연)** |
| | **dropped iterations** | **43** | **11,487** | 🔴 **+267배 폭증 (부하 처리 불능)** |
| | DB bookings 결과 | 정합성 OK | 정확히 10건 | ✅ 정합성 유지 |

### 🔍 1차 진단 및 핵심 발견
1. **정합성은 전 시나리오에서 완벽 보장**: 결제 통합 후에도 초과 판매 0건, 중복 멱등 필터링은 완벽하게 동작했습니다.
2. **Spike 부하 시 재앙적인 정체 발생**: 평시 부하(stress)나 극단적 연타(idempotency)에서는 병목이 부각되지 않았으나, 순간 폭증 트래픽(1000 TPS Spike) 상황에서 **처리량은 34% 폭락하고, 대기 지연(p95)은 100배가 늘어난 5.2초로 치솟아 시스템 마비 상태**에 이르렀습니다.

## 3. [원인 분석 & 개선 가설] 병목 원인 규명 및 최적화 전략 제안

### ⚠️ 재앙적 지연의 원인 분석
결제 통합 후, 1건의 예약을 처리하기 위해 애플리케이션이 감당해야 하는 연산 비용이 비약적으로 증가했습니다.

1. **DB 작업량의 6배 폭증 및 다중 트랜잭션 점유**:
   - 예약 1건당 DB 쓰기 작업이 **bookings 1건(기존) → bookings 1 + payments 2 + payment_histories 2 + point_transactions 1 = 총 6건**으로 늘어났습니다.
   - 트랜잭션 경계가 분리되면서 HikariCP 커넥션 점유 시간이 기하급수적으로 늘어났습니다.
2. **요청 대기 큐 정체 현상**:
   - 1000 TPS 동시 요청 유입 시, 제한된 WAS 스레드(Tomcat 200)가 모두 결제 처리 트랜잭션에 묶여 커넥션 풀(10개)을 획득하기 위해 길게 대기 큐를 형성했습니다. 이로 인해 후속 요청들이 줄줄이 대기하며 레이턴시가 5초대까지 누적 고갈되었습니다.

### ❌ 발견된 결정적 비효율: 검증 순서
기존의 비즈니스 검증 및 처리 흐름은 아래와 같았습니다.

```
[기존 흐름]
1. findExistingBooking() ──────> MySQL 멱등키 조회
2. existsById(userId) ─────────> MySQL 유저 존재 확인
3. getProductAndValidatePrice() > MySQL 상품 정보/가격 조회
4. reserveStock() ──────────────> Redis Lua 재고 차감 (실패 요청 990건은 여기서 다 거절됨)
5. processPayment() ────────────> 외부 PG 및 포인트 결제 처리
6. persistBookingAndPayments() ─> MySQL 최종 INSERT 저장
```

* **문제점**: 1000개의 동시 요청 중 **990개는 이미 재고 부족(SOLD_OUT)으로 거절될 운명**이었습니다. 그러나 이 990개의 실패할 요청들이 앞단에서 **유저 조회, 상품 조회 등 불필요한 DB SELECT 쿼리를 2,970회 선행 수행**하여 커넥션 풀을 조기에 바닥내고 있었습니다.

---

### 💡 개선 가설 수립: "Fail-Fast 재고 차감 최선두 배치"
검증 순서를 **"가볍고 가장 결정적인 필터링 역할을 하는 연산"** 순서로 재배치하기로 합니다.

```
[개선된 흐름]
1. 멱등 락 및 1차 검증 (Redis / PK 조회)  ← 매우 가벼움
2. reserveStock() (Redis Lua 재고 차감)   ◀ [최우선 배치] 990건은 여기서 1ms 만에 즉시 거절!
3. 사용자/상품 검증 (MySQL)              ◀ [성공한 10건만] DB 커넥션 소모
4. processPayment() (결제 처리)
5. persistBookingAndPayments() (최종 저장)
```

* **기대 효과**:
  - 거절될 990건에 대해 발생하는 **MySQL 조회를 완전히 0건으로 감소**시킵니다.
  - DB 커넥션 점유 시간을 비약적으로 단축시켜 HikariCP 풀의 회전율을 극한으로 끌어올립니다.
  - 비록 성공한 10건의 처리 자체 비용(PG + DB 6건)은 동일할지라도, 대기 큐의 정체 자체가 사라지므로 p95 Latency가 획기적으로 개선될 것입니다.

## 4. [개선 후 결과 비교] 최적화 적용 전후 성능 대조

수립한 가설에 따라 **우선순위 1 (검증 순서 최적화 - 재고 차감 우선)**을 소스 코드에 실제로 적용한 후, 동일한 1000 TPS Spike 시나리오를 재측정한 비교 데이터입니다.

### 📊 2차 측정 결과 (최적화 튜닝 적용 후 최종 비교)

| 시나리오 (Spike 부하) | 항목 | 1단계: 결제 없음 (Day 2) | 2단계: 결제 포함 (Day 3 - 튜닝 전) | **3단계: 검증 순서 최적화 (우선순위 1 적용 후)** | **튜닝 전 대비 개선폭** |
|---|---|---|---|---|---|
| **spike** (50→1000→50, 1.5분) | **평균 TPS** | 684 | 454 | **684.06** | ⚡ **+50.7% 처리량 증가** (목표치 정상 수렴) |
| | **p50 latency** | - | 3,756ms | **2.86ms** | ⚡ **-99.9% 감소** (실패 트래픽 실시간 즉각 거절) |
| | **p95 latency** | 50.6ms | **5,213ms** | **87.00ms** | ⚡ **-98.3% 폭감** (지연 현상 완벽 극복) |
| | **dropped iterations** | 43 | **11,487** | **49** | ⚡ **-99.6% 해소** (클라이언트 요청 유실 차단) |
| | DB bookings 결과 | 정합성 OK | 정확히 10건 | **정확히 10건** | ✅ **비즈니스 정합성 및 초과판매 0개 보장** |

> [!TIP]
> **성공 확정 트래픽(10건)에 대한 정밀 Latency 분포 (`expected_response:true`)**
> * **Median (p50)**: 47.91ms
> * **p95 Latency**: 322.43ms
> * **Max Latency**: 365.31ms
> * 복합 결제 및 다중 쓰기 비용을 완전히 부담하는 성공 트랜잭션 10건 역시 최대 360ms대의 쾌적한 속도로 완벽 소화되었습니다.

### 📢 최적화 지표 분석 및 성과
1. **가설의 성공적 실증**:
   - `reserveStock`을 앞당긴 덕분에 대다수의 실패 트래픽이 데이터베이스의 자원을 빌리지 않고 Redis 선에서 정리되었습니다.
   2. 이로 인해 DB 풀 정체 락이 풀리면서 p95 Latency가 **5.2초에서 87ms로 98.3% 폭감**했습니다.
3. **사실상의 요청 유실 제로화**:
   - 서버의 병목 정체가 해소되어 k6 부하 클라이언트가 요청 생성을 포기하고 버리던 `dropped_iterations`가 **11,487건에서 단 49건**으로 줄어들어, Spike 트래픽을 서버가 안전하게 감당해 냈습니다.
4. **완벽한 정합성 및 예외 원복(Saga) 신뢰성 유지**:
   - 재고가 선차감되는 구조로 개편된 상태에서도, 후행 예외(유저 검증 실패, 가격 불일치 등) 발생 시 레디스 재고를 원복해 주는 `compensate()` 안전망이 철저히 작동하여 초과 판매 0건, 최종 확정 예약 10건을 한 치의 오차 없이 완수했습니다.
