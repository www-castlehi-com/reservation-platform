## 4. 성능 비교 분석 및 개선 결과

### 📊 00시 폭증 (Spike 시나리오) 개선 효과 비교
- **p95 지연 시간**: `208.26ms` ➔ **`5.57ms` (97.3% 단축)**
- **p99 지연 시간**: `479.14ms` ➔ **`13.62ms` (97.1% 단축)**
- **요청 유실률**: `575건 드롭` ➔ **`0건 드롭` (완벽 방어)**

### 💡 기술적 해석 및 관찰 리포트
1. **메모리 레이어의 조기 차단 성공**: \
   `spike` 시나리오에서 1000 TPS의 트래픽이 몰려도 정적 데이터(상품 및 결제 수단) 조회가 Redis 캐시 상에서 처리되면서, RDBMS 커넥션을 빌리기 위한 대기 시간이 제거되어 속도 향상을 이루었습니다.
2. **단일 인스턴스의 물리적 한계점 포착**: \
   정적 조회를 캐싱했음에도 불구하고 `peak` 시나리오(1000 TPS 지속 부하)에서 p95 지연이 2.7초까지 튀고 5,369건의 드롭이 발생했습니다.\
   이는 DB 쿼리 병목이 아니라 단일 WAS(Tomcat) 프로세스가 가질 수 있는 자원(Thread Pool, OS File Descriptor 및 Network 가용 세션 부하)의 물리적 한계를 의미합니다.

   이 관찰 결과를 바탕으로, 다음 과제 요건인 2대 이상의 애플리케이션 서버 분산 인프라 및 Nginx 로드 밸런싱 구조를 적용하여 단일 서버의 하드웨어 한계를 극복하고 검증할 계획입니다.
