# Checkout API

## 1. 테스트 환경
- **인스턴스 구성**: 단일 로컬 인스턴스
- **대상 API**: `GET /checkout?productId=1`

## 2. 시나리오별 핵심 지표 요약

| 시나리오 | 목표 TPS | 실제 처리 TPS | p50 (중위값) | p95 (지연 임계) | p99 (최악 지연) | 에러율 | 총 처리량 (Iterations) |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **smoke** | 10 TPS | 10.02 TPS | 22.21 ms | 32.36 ms | 50.39 ms | 0.00% | 301 건 |
| **baseline** | 50 TPS | 50.00 TPS | 11.58 ms | 16.69 ms | 19.46 ms | 0.00% | 6,001 건 |
| **spike** | 가변 (최대 1000) | 558.37 TPS | 4.61 ms | 208.26 ms | 479.14 ms | 0.00% | 69,799 건 |
| **peak** | 1000 TPS | 993.15 TPS | 4.73 ms | 268.62 ms | 494.83 ms | 0.00% | 59,594 건 |

## 3. 상세 결과 분석 및 관찰 리포트

### ① smoke & baseline 시나리오 (평시 부하)
- **분석**: 평시 요건인 50 TPS 성능 환경에서는 p95 지연 시간이 16.69ms로 매우 안정적입니다. 에러율 역시 0%를 기록하며 기본적인 데이터 소화 능력을 검증했습니다.

### ② spike & peak 시나리오 (00시 폭증 부하 및 피크 한계)
- **병목 구간 식별**:
    - `peak` 시나리오에서 목표인 1000 TPS에 근접한 **993.15 TPS**까지 소화해내며 에러율 0%를 유지한 것은 고무적입니다.
    - 하지만, 대량의 트래픽이 RDBMS로 직접 쏠리면서 멱함수 분포에 따른 꼬리 지연(Tail Latency) 현상이 관찰되었습니다. **p95 지연 시간이 268.62ms**, **p99 지연 시간이 494.83ms**까지 치솟았으며, 최대 지연 시간(`max`)은 **1098.27ms**로 1초를 돌파했습니다.
    - 이로 인해 목표한 처리량을 제때 소화하지 못하고 밀려나 버려진 **`dropped_iterations`가 `spike`에서 575건, `peak`에서 407건 발생**했습니다. 이는 데이터베이스 커넥션 대기열(HikariCP Queue)이 병목을 일으키기 시작했다는 명확한 증거입니다.

## 4. 분석
현재는 에러가 없으나 2대 이상의 분산 환경으로 확장되거나, 실제 DB 쓰기 트래픽(`POST /bookings`)과 같은 커넥션을 공유하게 될 경우 RDBMS 경합으로 인해 시스템이 완전히 붕괴할 위험이 큽니다.
