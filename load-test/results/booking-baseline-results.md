## 1. 테스트 환경
- **인스턴스 구성**: 단일 로컬 인스턴스 (스프링 부트 1대 + MySQL)
- **대상 API**: `POST /bookings`
- **사전 조건**: 1번 상품의 최초 제한 재고 = **10개**

## 3. 시나리오별 상세 기술 분석 및 관찰 리포트

### 🛑 ① stress 시나리오: 동시성 경합 및 지연 발생
- **현상**: 락이 전혀 없는 단순 조회(`findById`) 구조임에도 불구하고 응답 지연 시간이 **p95 기준 349.80ms**까지 밀렸습니다.
- **원인 분석**: 100개의 요청이 동시에 MySQL `products` 테이블의 동일한 행(1번 상품)을 갱신(`UPDATE products SET total_stock = ...`)하려고 들이닥치면서, 데이터베이스 내부적으로 **로우 락(Row Lock) 경합 및 디스크 I/O 대기열 병목**이 심화되었습니다.
- **정합성 붕괴**: 자바 레이어의 `if (product.getTotalStock() <= 0)` 조건문 검증 시점과 실제 쿼리가 날아가 디스크를 치는 시점 사이에 시차(**Race Condition**)가 발생하여, **실제 한정 재고인 10개를 초과한 판매(Over-selling)** 현상이 유도되었습니다.

### 💥 ② idempotency 시나리오: 애플리케이션 멱등성 완전 붕괴
- **현상**: 동일 유저가 동일 멱등키로 100회 연타를 날렸을 때, k6 지표상 성공률 0% / 에러율 100%가 찍히며 전량 실패했습니다. 또한 **재고가 전혀 차감되지 않는 기현상**이 관찰되었습니다.
- **원인 분석**: 애플리케이션 내부에서 중복 요청을 사전에 걸러내는 멱등성 필터링 메커니즘이 존재하지 않아, 100개의 연타 요청이 전부 조건문을 통과하여 `bookingRepository.saveAndFlush()` 단계까지 진입했습니다.
- **롤백 메커니즘의 약점**: 뒤늦게 MySQL 인프라의 `idempotencyKey UNIQUE` 제약조건벽에 부딪히면서 `DataIntegrityViolationException`이 무더기로 쏟아졌고, 스프링 부트의 **`@Transactional` 규칙에 의해 해당 트랜잭션 전체가 통째로 롤백(Rollback)** 되었습니다.

### 🌋 ③ spike 시나리오: 00시 1000 TPS 폭증 시의 대참사
- **현상**: 37,569건의 무차별 폭격 상황에서 k6가 인지한 성공 응답(`expected_response: true`)은 **단 11건**이며, 나머지 99.97%는 서버가 던진 비즈니스 예외 혹은 DB 연결 타임아웃 예외로 실패 처리되었습니다.
- **아키텍처적 시사점**: 제한 재고가 10개임에도 k6 지표 자체에서 이미 11개의 성공이 떨어져 **초과 판매가 실측**되었습니다. 더불어 분산 트랜잭션 원자성이 없어 톰캣 스레드 풀이 마비되고 디스크 부하가 임계치를 넘는 아웃 오브 컨트롤 상태를 보여줍니다.

## 4. 최종 결론

**"디스크 기반 RDBMS의 'Read-Check-Write' 아키텍처와 DB UNIQUE 제약조건에 전적으로 의존하는 정합성 제어는 1000 TPS 대규모 트래픽 환경에서 정합성과 성능을 동시에 잃고 시스템을 완전히 파괴한다."**는 사실을 지표로 명확히 증명했습니다.

이를 해결하기 위해 다음 가설을 검증합니다.
> **[가설]** \
> 무거운 RDB 비관적 락으로 스레드를 블로킹하는 대신, 초고속 인메모리 레이어에서 싱글 스레드로 원자적(Atomic) 연산을 수행하는 **Redis의 `DECR` 명령어로 재고 차감 책임을 전면 이전**한다. \
> 이를 통해 1000 TPS의 폭격이 오더라도 대문 밖(Redis 레벨)에서 정확히 10개만 통과시키고 나머지를 초고속 컷(`SOLD_OUT`)하는 인프라 가용성 확보 능력을 2차 측정한다.
