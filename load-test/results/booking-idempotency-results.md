## 1. 테스트 환경
- **인스턴스 구성**: 단일 로컬 인스턴스 (스프링 부트 1대 + MySQL + Redis 7-Alpine)
- **대상 API**: `POST /bookings`
- **구현 스펙**: Redis Lua Script(재고 선점) + 3-Tier 멱등성(Application DB 조회 ➔ Redis SETNX 분산 락 ➔ DB UNIQUE 제약조건) + Saga 패턴 기반 보상 트랜잭션
- **사전 조건**: 1번 상품의 제한 재고 = **10개** (`redis-cli SET stock:product:1 10`)

## 3. 핵심 정합성 검증 결과 (무결성 검증 완료)

### 🛡️ ① 재고 정합성 및 초과 판매 방어 (`stress` 시나리오)
- **실측 데이터**: 100개의 인스턴스가 동시에 들이닥쳤을 때, `is valid sales response` 체크를 통해 정확히 10건의 성공과 90건의 정상 예외(SOLD_OUT)가 분리되었습니다.
- **DB 검증**: `SELECT COUNT(*) FROM bookings` 수행 결과 **정확히 10건**의 영수증이 적재되었으며, 채번 충돌 없이 고유한 예약 번호가 부여되었습니다.
- **아키텍처적 의의**: Redis Lua 스크립트를 활용해 `GET ➔ 비교 ➔ DECR` 연산을 싱글 스레드 내부에서 원자적으로 처리함으로써, 1단계에서 발생했던 **초과 판매(Over-selling)가 제거**되었습니다.

### 🔄 ② 연타 공격 방어 및 조기 차단 효율 (`idempotency` 시나리오)
- **실측 데이터**: 3명의 유저가 동일한 멱등키로 1초 동안 100번씩 미친 듯이 연타(총 300요청)를 날렸으나, `is processed or blocked gracefully` 검증 **Pass 100% (300건 전체 성공)**를 달성했습니다.
- **DB 검증**: `bookings` 테이블 확인 결과, 중복 생성 없이 정확히 3건(고유 키당 1건)만 완벽히 저장되었습니다.
- **성능 반전**: 1단계에서는 동일 연타가 DB까지 밀고 들어와 집단 500 에러와 롤백 헬게이트를 열었으나, 3단계에서는 **p95 지연 시간이 단 4.92ms**로 극도로 가볍게 처리되었습니다.
- **기술적 고찰**: 이는 2차 방어선인 Redis `SETNX` (In-Flight 락)과 1차 방어선인 DB 조회가 연타 트래픽의 99%를 무거운 DB 트랜잭션 진입 전에 조기 차단(Early Exit)하여 스레드 풀을 완벽하게 보호해 냈음을 증명합니다.

## 4. 고부하 성능 특성 및 단일 인스턴스 임계점 분석

### 📈 Checkout API(조회) 대비 성능 트레이드오프 분석
- 앞서 진행한 `Checkout API` 캐싱 부하 테스트 당시 `spike` 시나리오의 p95 지연 시간은 **5.57ms**였습니다. 반면, 본 `Booking API` 최종형의 `spike` 전체 평균 p95 지연 시간은 **50.57ms**로 측정되었습니다.
- **원인 분석**: `Checkout`은 순수하게 인메모리(Redis) 조회 후 즉시 빠져나가는 구조인 반면, `Booking`은 Redis 가예약 선점 성공 후 실제 디스크 기반 RDBMS(MySQL)에 무거운 영수증 쓰기(`INSERT`) 및 트랜잭션 동기화(Flush)가 동반되는 '쓰기 중심 트래픽'이기 때문에 약 10배의 시차가 발생하는 것은 자연스러운 고부하 인프라의 특성입니다.

### ⚠️ 단일 인스턴스 환경의 하드웨어 임계점 포착
- **목표치 미달**: 1000 TPS 폭증을 목표로 설정했으나, 실제 평균 처리량은 **684.17 TPS**에서 수렴했습니다.
- **드롭 레이트 발생**: 테스트 도중 k6 엔진 단에서 제때 요청을 밀어 넣지 못하고 버려진 **`dropped_iterations`가 43건 발생**했습니다.
- **결론**: 이는 애플리케이션 소스 코드나 DB 락의 병목이 아니라, **단일 톰캣 서버가 네트워크 디스크 I/O와 스레드 컨텍스트 스위칭을 동시에 소화할 수 있는 물리적 자원의 임계점**에 도달했음을 뜻합니다. 성능을 1000 TPS 이상으로 선형 확장하기 위해서는 **'애플리케이션 서버 다중화(Scale-out) 및 Nginx 로드 밸런싱' 인프라 확장이 반드시 필요함**을 의미합니다.

## 5. 재고 관리의 진실의 원천 (Source of Truth) 격리화

- 본 테스트 종료 후, MySQL 상품 테이블의 `products.total_stock`은 여전히 최초 시드 값인 **10**으로 유지되는 반면, Redis의 `stock:product:1` 캐시 값은 **0**으로 완벽하게 잠겨 있습니다.
- **의도된 아키텍처 아웃풋**:
    - **재고의 진실의 원천(Source of Truth) = Redis String 캐시**
    - **RDB `total_stock` = 상품 오픈 시점의 초기 메타 정보**
- **도입 사유**: 만약 선착순 폭격이 올 때마다 DB 상품 테이블의 특정 Row를 실시간으로 `UPDATE` 치게 만들면 디스크 경합으로 시스템이 붕괴합니다. 따라서 정적 마스터 데이터는 메타 정보로 두고, **실시간 차감 비즈니스는 메모리 레이어로 완전히 격리**하여 RDB 단일 로우 핫스팟 경합을 근본적으로 우회했습니다. (향후 트래픽이 안정화되는 시점에 배치 배포를 통해 Redis 잔여 재고를 RDB에 일괄 동기화하는 Write-Behind 기법을 사용하는 컨셉입니다.)
