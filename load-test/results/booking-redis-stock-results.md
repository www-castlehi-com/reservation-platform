## 3. 핵심 기술 분석 및 아키텍처적 고찰

### 🌋 spike 시나리오: 분산 시스템의 정합성 균열 실측
- **현상**: 37,302건의 고동시성 폭격 환경에서 **Redis 재고=0, RDB 상품 재고=10, 최종 예약 영수증(`bookings`)=10건**이라는 인프라 간 데이터 불일치 상태가 발생했습니다.
- **원인 분석**: 싱글 스레드로 원자적 연산을 보장하는 Redis 메모리 레이어의 도입 덕분에, 1단계의 가장 치명적인 문제였던 **'초과 판매(Over-selling)'는 완벽하게 차단**하는 데 성공했습니다.
- **한계점 및 리스크**: 그러나 'Redis 재고 차감'과 'RDBMS 예약 확정'이 하나의 원자적 트랜잭션 경계로 묶이지 못하는 **분산 시스템의 원자성 결여**가 관찰되었습니다. 만약 후속 DB 저장 과정에서 예외가 발생하여 트랜잭션이 취소되더라도 이미 차감된 Redis 재고는 자동으로 롤백되지 않고 유실되는 치명적인 데이터 격차가 존재합니다.

### 💥 idempotency 시나리오: 무거운 DB 레벨 롤백 의존성 유지
- **현상**: 동일 유저 3명이 각 100회의 따닥 연타를 날렸을 때, 애플리케이션 단에서 차단되지 못하고 전량 DB UNIQUE 제약조건 벽에 충돌하여 100% 롤백 처리되었습니다.
- **리스크**: 인프라 자체의 무결성은 지켜졌으나(성공 0건), 만약 이 비즈니스 로직 중간에 **외부 PG사 결제 API 호출이 연동되어 있다면, 예약은 실패했으나 유저의 카드는 100번 중복 긁히는 대형 장애**가 유발됨을 뜻합니다.

## 4. 향후 개선 로드맵 명분 확립

본 기능 개선을 통해 성능 효율 향상과 초과 판매 방어라는 성과를 거두었으나, "분산 캐시 레이어와 디스크 RDB 레이어 간의 상태 동기화 및 3-Tier 멱등성 조기 차단"이 결합되지 않으면 완벽한 선착순 시스템을 보장할 수 없음을 수치로 증명했습니다.
